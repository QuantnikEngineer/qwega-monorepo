1. Architecture Overview

System Description

The QUANTNIK SDLC Multi-Cloud CI/CD architecture enables automated build and deployment of microservices to three cloud providers simultaneously. When code is pushed to a monitored branch, the system automatically triggers a pipeline that builds the application and deploys it to GCP Cloud Run, Azure Container Apps, and AWS ECS Express Mode in parallel.

Key Components

Component Type Description
Git Repositories Source 18 QUANTNIK SDLC microservice repositories hosted on Harness Git
Webhook Triggers Automation Branch-level triggers on push events (main, develop, release/*)
Scheduled Triggers Automation Cron-based triggers for nightly builds (2:00 AM daily)
QUANTNIK SDLC Orchestrator Pipeline Parent pipeline that coordinates multi-cloud deployment
CloudRun Worker Pipeline GCP Builds, scans, deploys to Cloud Run
ACA Worker Pipeline Azure Builds, scans, deploys to Container Apps
ECS Express Worker Pipeline AWS Builds, scans, deploys to ECS Express Mode

3. Trigger Configuration

Webhook Trigger (On Commit)

Each repository has a dedicated webhook trigger that fires when code is pushed to monitored branches. The trigger extracts the branch name and passes it to the orchestrator pipeline. 

# Example: Webhook Trigger for quantnik-brd repository
trigger:
  name: Trigger_WegaBrd_OnCommit
  pipelineIdentifier: WEGASDLCOrchestrator
  source:
    type: Webhook
    spec:
      repoName: quantnik-brd
      actions: [Push]
      payloadConditions:
        - key: targetBranch
          operator: Regex
          value: ^(main|develop|release/.*)$
  inputYaml: |
    pipeline:
      variables:
        - name: selectedRepo
          value: quantnik-brd
        - name: deployTarget
          value: all
        - name: branchwegaBrd
          value: <+trigger.branch>

Branch Pattern Reference & Release Strategy

PatternExamplePurposeAuto-Deploy To
developdevelopIntegration branch (main development)Dev only
feature/*feature/new-authFeature branches (cut from develop)Dev only
release/*release/quantnik-04292026Weekly release (cut from develop)QA → Stage → Prod

📅 Release & Branching Strategy

1. Development Flow: Developers work on feature/* branches cut from develop. Both feature and develop branches deploy to Dev environment only.

2. Weekly Release: Every week, a release/* branch is cut from develop. This release branch is deployed through the promotion pipeline: QA → Stage → Prod.

3. Approval Gates: Release branches require approvals between QA → Stage and Stage → Prod (14-day windows).

4. QUANTNIK SDLC Orchestrator

🎯 Key Feature: deployTarget = "all"

When deployTarget is set to "all", the orchestrator invokes all three worker pipelines (CloudRun, ACA, AppRunner) in parallel, enabling simultaneous deployment to all cloud providers from a single trigger event.

Stage Conditions

# Each worker stage evaluates this condition:
condition: <+pipeline.variables.deployTarget> == "cloudrun" || <+pipeline.variables.deployTarget> == "all"
condition: <+pipeline.variables.deployTarget> == "aca" || <+pipeline.variables.deployTarget> == "all"
condition: <+pipeline.variables.deployTarget> == "ecsexpress" || <+pipeline.variables.deployTarget> == "all"

⚡ AWS Amazon ECS Express Mode

🚀 Simplified Container Deployment

Amazon ECS Express Mode reduces the complexity of deploying containerized applications by providing sensible defaults and automating the configuration of supporting AWS services. Instead of managing configuration parameters across multiple services, Express Mode requires only three things to get started:

  • Container Image - Your Docker image in ECR
  • Task Execution Role - IAM role for ECS tasks
  • Infrastructure Role - IAM role for infrastructure management

Key Benefits

⚡ Simplified Deployment

Deploy production-ready containerized applications with sensible defaults. ECS Express Mode orchestrates all necessary infrastructure automatically.

🔧 Auto-Configured Infrastructure

Automatically provisions: Fargate service, Load Balancer with SSL/TLS, auto scaling policies, CloudWatch monitoring, and VPC networking.

💰 Cost Optimization

Shares Application Load Balancers across multiple Express Mode services using the same networking configuration to reduce costs.

ECS Express Mode Features

Feature Description
Deployment Complexity Low - Only 3 required inputs (image, task role, infrastructure role)
Infrastructure Access Full visibility & direct access to all underlying AWS resources
Load Balancer Application Load Balancer with SSL/TLS auto-configured
Auto Scaling Configurable scaling policies based on utilization or traffic
Networking Your VPC with public or private HTTPS applications
Compute Fargate-based ECS service (serverless containers)
CI/CD Integration AWS CLI, CloudFormation, Terraform, GitHub Actions

Pipeline Integration

# ECS Express Mode Deployment Step

Stage: Deploy_ECS_Express

1. Prerequisites
   ├── Container image pushed to ECR
   ├── Task execution role configured
   └── Infrastructure role with ECS permissions

2. Create/Update Express Service
   # Using AWS CLI
   aws ecs create-express-service \
     --service-name <+pipeline.variables.serviceName> \
     --image <+pipeline.variables.imageUri> \
     --task-execution-role-arn <+variable.taskExecutionRole> \
     --infrastructure-role-arn <+variable.infrastructureRole> \
     --public-access true \
     --port 8080

3. Output
   ├── Unique HTTPS URL (auto-generated)
   ├── Fargate tasks running
   ├── ALB with SSL/TLS configured
   └── CloudWatch logs & metrics enabled

4. Health Check
   └── Verify service URL responds with HTTP 200

📋 When to Use ECS Express Mode

✅ Ideal For:

  • Stateless web applications & APIs
  • Rapid prototyping with production defaults
  • Teams wanting more control than App Runner
  • Platform teams enabling self-service deployments

❌ Consider Alternatives For:

  • Complex multi-container deployments
  • Stateful applications requiring EBS
  • Custom task definitions needed
  • EC2 launch type requirements

🔙 Rollback Strategy

# Rollback Options

1. Image Tag Rollback
   - Re-run pipeline with previous imageTag
   - executionMode: deploy_only
   - imageTag: <previous-execution-id>

2. Cloud Run Revision Rollback
   - Use Cloud Run console/CLI
   - Route traffic to previous revision
   - Instant rollback capability

3. Git-based Rollback
   - Revert commit in source repo
   - Trigger new build from reverted code
   - Full audit trail maintained
