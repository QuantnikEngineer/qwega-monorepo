---
name: tech-debt-check
description: Statically analyses the project codebase for technical debt — duplication, dead code, complexity, large files, TODO/FIXME backlog, test gaps, outdated patterns, weak typing, deprecated APIs, magic literals, performance smells, and dependency hygiene — ranks findings by maintainability cost × git change frequency (hotspots first), applies only the very-low-risk automated fixes (unused imports, var→const, ==→===, unreachable code, sort imports, magic-number extraction), and publishes a structured debt report to Confluence with the URL printed at the end. Each finding gets a severity, a category, the file/lines, an evidence snippet, an "auto-fixed" / "needs-refactor" status, and a row in the published report.
---

When invoked, run the audit end-to-end. **Do not ask the user clarifying questions** mid-run unless a step truly cannot proceed (e.g. no project cwd). Take the reasonable default at every fork and report what you decided.

This skill supports the same two Atlassian MCP shapes as `vulnerability-check`:
- **Shape A — wega2 stdio:** `mcp__Confluence__conf_get/post/...`. Site URL is `https://<ATLASSIAN_SITE_NAME>.atlassian.net`.
- **Shape B — claude.ai-managed:** `mcp__claude_ai_Atlassian__createConfluencePage`, `getConfluenceSpaces`, etc.

If neither shape is loaded, fall back to writing `<project>/docs/tech-debt-<timestamp>.md` and print that local path as the final output line.

**This skill is the maintainability/quality counterpart to `vulnerability-check`.** Anything that's a *security* exploit (secrets, injection, weak crypto, dep CVEs, auth holes) belongs in `vulnerability-check` — do **not** duplicate those categories here. If you spot a security issue while scanning for debt, mention it in the report's "Cross-link" section with a suggestion to run `/vulnerability-check`.

---

## Phase 0 — Scope

Decide what to scan.

- **Always:** the project cwd.
- **Also:** every `additionalDirectories` (configured repos).
- **Skip:** `node_modules`, `.git`, `dist`, `build`, `.next`, `.turbo`, `.cache`, `coverage`, `__pycache__`, `.venv`, `venv`, `target`, `.gradle`, `out`, `tmp`, generated lockfiles, anything matching `*.min.js`, `*.bundle.js`, `*.generated.*`, snapshot files.

Detect languages by extension: `.js .jsx .mjs .cjs .ts .tsx .py .java .go .rb .php .cs .rs`. Note which are present — some categories are language-specific (e.g. TS-only "weak typing").

Decide if `git` is available (`git rev-parse --is-inside-work-tree`). The hotspot ranking depends on it; without git, fall back to file mtime as a proxy and note this in the report metadata.

Print a one-line scope header:

```
Tech-debt scan: <cwd> (+ N extra repos) — <K> source files across <langs> · git=<yes|no>
```

---

## Phase 1 — Collect signals

Run every category below. Use `Grep`, `Read`, `Bash` (for `git`, optional CLI tools) and structured analysis. None of these should be a security category — keep that boundary.

### A. Hotspot data (drives the final ranking)

For each source file in scope, capture:

- `loc` — line count (`wc -l` or read+split).
- `commits_90d` — `git log --since="90 days ago" --oneline -- <file> | wc -l`. Files with `commits_90d ≥ 5` are **hotspots**.
- `last_modified` — `git log -1 --format=%ct -- <file>`.
- `authors_90d` — `git log --since="90 days ago" --format=%an -- <file> | sort -u | wc -l`. High author count + high commit count = "shared hot file" (worse to leave debt in).

If git isn't available, skip these and use mtime/loc only.

### B. Duplication — debt category `duplication`

- **Block-level:** detect runs of 6+ consecutive non-blank lines that appear identically (after trimming) in 2+ places across files. Use a rolling hash over normalised tokens (collapse whitespace, drop string contents to `"…"`, drop number literals to `N`).
- **Function-level:** within a file, detect two functions whose bodies differ by ≤ 3 normalised tokens — likely copy-paste.
- Skip duplication inside obvious generated code (look for `// THIS FILE IS GENERATED`, `# autogenerated`, `// @ts-nocheck` blocks).

Per finding: list every location of the duplicate group + a snippet of the first occurrence + the suggested abstraction (extract function / shared util).

### C. Dead code — `dead-code`

- **Unused imports:** import statement whose binding(s) are never referenced in the file body. Skip side-effect imports (`import './polyfills'` / `import 'reflect-metadata'`).
- **Unused exports:** named export with **zero** references across the in-scope files (use `Grep` for the symbol; require a real usage, not the export site itself). Skip public-API entry-points: `index.{js,ts}` at package root, files listed in `package.json#exports`/`main`/`bin`, anything in `pages/`, `app/`, `routes/`, framework convention dirs.
- **Unreachable code:** statements that follow `return`/`throw`/`process.exit`/Python `raise`/`sys.exit` in the same block.
- **Large commented-out blocks:** ≥ 4 consecutive lines beginning with `//`/`#` that contain code-like tokens (`= ;`, `function `, `def `, `class `, `return `). Skip license headers and known-shape doc-banners.

### D. Complexity — `complexity`

For each function/method:

- **Length:** body LOC > 50 → flag; > 100 → high severity.
- **Cyclomatic-ish:** count branching keywords (`if`, `else if`, `case`, `catch`, `for`, `while`, `&&`, `||`, `?` ternary). Threshold `> 12` → medium; `> 25` → high. (Approximate, not strict McCabe.)
- **Nesting depth:** track brace/indent depth. `> 4` levels → medium; `> 6` → high.
- **Parameter list:** > 5 positional params → medium; > 8 → high. Boolean traps (multiple `bool` params, or `bool` params next to each other without names at call sites) → medium.

### E. File size — `file-size`

- Source files > 500 LOC → medium.
- Source files > 1000 LOC → high.
- Test files allowed up to 1000 LOC; flag above.
- "God" files: ≥ 1 file accounting for > 30% of LOC in its directory → medium even below the absolute threshold.

### F. TODO / FIXME / HACK backlog — `todo-backlog`

- Grep for `\b(TODO|FIXME|HACK|XXX|TBD|TEMP|REMOVEME)\b`.
- For each match, run `git blame -L <line>,<line> -- <file>` to get the commit timestamp and author.
- Rate by age:
  - > 12 months → high (likely fossilised).
  - 3–12 months → medium.
  - < 3 months → low (still hot).
- Aggregate counts in the report; list the top 20 oldest individually.

### G. Test gaps — `test-gap`

- For each non-test source file, check if a sibling test exists:
  - JS/TS: `<base>.test.{js,ts,jsx,tsx}` or `<base>.spec.*` or in `__tests__/<base>.*` or `tests/<base>.*`.
  - Python: `tests/test_<base>.py` or `<base>_test.py`.
  - Go: `<base>_test.go`.
- Files with no test AND > 80 LOC AND **changed in the last 90 days** → high (you're shipping untested change).
- Files with no test AND > 80 LOC AND not recently changed → medium.
- Files < 80 LOC AND no test → low.

(Coverage-percentage signals — if a `coverage/lcov-report` or `coverage.json` is present, parse it and use real coverage numbers instead of the heuristic.)

### H. Outdated patterns — `outdated-pattern`

- **JS/TS:** `var ` declarations, `==`/`!=` instead of `===`/`!==`, `arguments` in arrow-eligible callbacks, `function(` as a callback parameter where arrow would work, Promise constructor anti-pattern (`new Promise((res) => { somePromise.then(...).then(res) })`), `.then().then().then(` chains > 3 deep when await could replace.
- **TS:** `as any`, `: any`, `// @ts-ignore`, `// @ts-nocheck`, missing return types on exported functions, `Function` type usage, `Object` type usage.
- **Python:** `% formatting` instead of f-strings, mutable default args (`def f(x=[])`), `except:` bare clauses, `print(` for logging in library code, `from x import *`.
- **React:** `componentWillMount`/`componentWillReceiveProps`/`componentWillUpdate`, class components in a hooks-heavy codebase (count and only flag if hooks/class ratio > 80% hooks), `bind(this)` in render, inline functions in render props.
- **Java:** raw types (`List` without `<T>`), `new Date()` in business logic.
- **Go:** ignored errors (`_ = call()` returns where error was ignored), `panic(` in library code.

### I. Magic literals — `magic-literal`

Within each file, count occurrences of each:

- Numeric literal `> 1` that appears `≥ 4` times (skip `0`, `1`, `-1`, common indices like `2`/`3` unless they appear in business arithmetic).
- String literal of length ≥ 6 chars appearing `≥ 3` times (skip URLs in tests, JSON keys in serialisation code where they're inherently strings).

Suggest extraction to a named const at top of file or a `constants.{js,ts,py}` module if one exists.

### J. Performance smells — `perf`

- `await` inside `for`/`for..of`/`while` loop where the iterations don't depend on each other (typically `await` of separate http calls / db reads). Suggest `Promise.all`.
- Sync fs (`fs.readFileSync`/`fs.writeFileSync`) inside an async request handler.
- Manual `JSON.parse(JSON.stringify(obj))` for deep clone in hot paths (suggest `structuredClone`).
- N+1 query patterns: loop iterating over results, calling a `find`/`get`/`query` inside.
- React: `useEffect` with no deps array creating subscriptions / setting state on every render.

### K. Documentation gaps — `docs-gap`

- **Exported** functions/classes (the file's public surface) with **no preceding** JSDoc/TSDoc/docstring/Go-doc comment AND non-trivial body (≥ 10 LOC):
  - JS/TS: missing `/** ... */` immediately above an `export function`/`export class`/`export const = (` definition.
  - Python: function with no triple-string docstring as first statement.
  - Go: exported (capitalised) func with no preceding `// FuncName ...` line.
- Severity: low (info if private).

### L. Dependency hygiene — `dep-hygiene` (not a CVE category — those live in vulnerability-check)

- **Unused deps:** Try `npx --no-install depcheck --json` from the project root if `package.json` is present. Parse "dependencies" (not "missing"/"devDependencies"). If `depcheck` isn't installed, skip the precise check and instead grep for `require('<dep>')` / `from '<dep>'` for each entry in `dependencies` — anything with zero matches is suspect.
- **Outdated majors:** `npm outdated --json` (silently skip if absent). List packages where `current` and `latest` differ in **major** version. Severity medium.
- **Stale package age:** packages whose installed version was published > 3 years ago (`npm view <pkg>@<ver> time` — costly; do this only for top-level deps listed in `package.json`, not transitive).
- **Python / Go / Rust:** equivalents only if the tooling is on PATH; otherwise note "skipped — install pip-outdated / go-mod-outdated / cargo outdated for fuller coverage".

### M. Architecture drift — `arch-drift`

- **Circular imports:** build a per-file import graph for the in-scope language(s); flag any cycle. Severity high.
- **Layer violations:** if directory naming suggests layers (`ui/`, `pages/`, `components/` vs `db/`, `data/`, `repos/`, `infra/`), flag imports that bypass typical boundaries (e.g. UI importing DB directly). Heuristic; severity low unless project has `CLAUDE.md` or `ARCHITECTURE.md` stating rules — then medium.
- **Growth drift:** files that exceeded their previous-month LOC by > 40% AND now exceed Phase-1-E thresholds — note as "rapid growth, refactor before it cements".

## Phase 2 — Rank and build the findings list

For each finding, produce a structured entry:

```
- id: DEBT-001
  severity: Critical | High | Medium | Low | Info
  category: duplication | dead-code | complexity | file-size | todo-backlog |
            test-gap | outdated-pattern | magic-literal | perf | docs-gap |
            dep-hygiene | arch-drift
  file: <relative path>
  line: <number or range>
  description: <one paragraph: what's wrong, why it costs maintenance time>
  evidence: |
    <5–10 lines of context>
  fix-strategy: <one sentence on how to fix it>
  fixable: yes | needs-refactor
  hotspot_score: <see formula>
```

### Ranking formula

`hotspot_score = severity_weight × (1 + commits_90d) × (1 + authors_90d / 3)`

Severity weights: Critical 5, High 3, Medium 2, Low 1, Info 0.5.

Sort findings by `hotspot_score` descending so the *most-trafficked* debt floats to the top. Without git, set `commits_90d = authors_90d = 0` and the order collapses to severity-only.

### Severity rubric (debt-specific)

- **Critical:** circular import that already breaks production builds; a hotspot file (commits_90d ≥ 10) with zero test coverage; > 1000 LOC growing > 40%/month.
- **High:** > 500 LOC files; cyclomatic > 25; dead-code blocks > 100 LOC; missing test on file changed in the last 30 days; TODO older than 12 months.
- **Medium:** duplication groups across 3+ locations; complexity 12–25; nesting > 4; `as any` in TS in non-leaf modules; outdated major versions; magic literals occurring ≥ 4 times.
- **Low:** missing docstrings on exported functions; magic numbers occurring 3 times; var/== instead of let/===; minor outdated patterns.
- **Info:** style nits — boolean traps, inline-function-in-render, etc.

### Mark `fixable: yes` only for these safe automated fixes

The agent will apply these in Phase 3:

1. **Unused imports** — remove the import line.
2. **`var` → `const`/`let`** — `const` when never reassigned, else `let`.
3. **`==`/`!=` → `===`/`!==`** when both operands appear to be the same JS type (literal vs variable of inferred type). When unsure, mark `needs-refactor`.
4. **Unreachable code after `return`/`throw`** — delete it (and any blocks unreachable as a consequence).
5. **Import order / sort imports** — alphabetise within their original group (built-ins / external / internal), preserving comment headers.
6. **Magic-number → named const** when a number > 1 appears ≥ 4 times in a single file: introduce `const SOME_NAME = N;` at the top of the file (or in the existing constants section) and replace all occurrences. Use a name derived from surrounding identifiers; if the agent can't pick a clear name, mark `needs-refactor`.
7. **Sort `package.json` keys alphabetically within `dependencies` / `devDependencies`** — only if `package.json` exists. Don't touch versions.

Everything else is `fixable: needs-refactor` — it goes in the report but is not auto-patched.

Print the compact summary:

```
TECH-DEBT SCAN — <project>
Scanned <K> files in <ms>ms

| # | Severity | File:Line                | Category        | Hotspot | Issue                          |
|---|----------|--------------------------|-----------------|---------|--------------------------------|
| 1 | High     | src/router.js:1–612      | file-size       | 9.0 ×   | 612-line file, 14 commits/90d  |
| 2 | High     | src/auth.js:200–340      | complexity      | 8.5 ×   | Cyclomatic 31, depth 7         |
| ...

TOTAL: <N> findings — <c> Critical, <h> High, <m> Medium, <l> Low, <i> Info
Hotspot top-5 files: <list>
```

## Phase 3 — Apply the safe automated fixes only

Iterate `fixable: yes` findings, highest `hotspot_score` first. For each:

1. Read the affected file (smallest window that gives context).
2. Construct the patched version.
3. Apply via `Edit` (single-occurrence unambiguous) or `Write` for fully rewritten files (rare).
4. Run a syntax sanity check if available (`node --check`, `python -m py_compile`, `tsc --noEmit` if a `tsconfig.json` is present). On failure, revert and re-classify as `needs-refactor`.
5. Capture `before`/`after` snippets for the report.

When extracting a magic literal, place the new `const` immediately after any existing imports/header and before the first declaration. Name with `UPPER_SNAKE` for numeric constants, normal `camelCase` for shared strings. If a constants file already exists at `src/constants.{js,ts,py}` or similar, prefer adding there and importing.

Do **not** auto-fix any of these (always `needs-refactor`):

- Duplication consolidation (the abstraction shape is a human call).
- Function splitting for complexity (style-dependent).
- Type tightening (`any` → specific) — needs runtime knowledge of values.
- Deprecated API replacement (might subtly change behaviour).
- Dependency major-version bumps (covered by `vulnerability-check` for CVE-driven bumps; non-CVE majors are explicit human decisions).
- Anything that touches public API signatures.

If applying a fix would clobber uncommitted user edits in the file (check with `git diff --quiet -- <file>`; non-zero exit = dirty), skip that fix and log `status: skipped-dirty` for the report.

---

## Phase 4 — Verify

Re-run the cheap regex/text scans for each fixable category (skip the dep audit re-run). Diff against the original list:

- **Resolved:** finding ids no longer matching.
- **Remaining:** still-matching findings — these become `status: still-debt` in the report.
- **Introduced-by-fix:** any new pattern that appeared because of an edit — extremely rare, but check.

---

## Phase 5 — Publish the report to Confluence

**Target space is fixed by `wega.json` when present.** `Read` `.claude/wega.json` at the project cwd. If it has `atlassian.confluenceSpaceKey` or `confluenceSpaceId`, that is the **only** allowed target. Do **not** fall back to "first personal space" / `getConfluenceSpaces` when the sidecar has a value.

Title: `<Project> — Tech-Debt Report — <YYYY-MM-DD HH:mm>`.

Body (storage HTML for Shape A, markdown for Shape B with `contentFormat: "markdown"`):

1. **Executive summary** — total findings, breakdown by severity, by category; how many auto-fixed; the top 5 hotspot files with their commit-frequency stats; one-paragraph "what this means for the team".
2. **Severity & category legend** — small table mapping rubrics; one sentence per category explaining the impact.
3. **Hotspot heat-map** — table of the top 20 files by `commits_90d` with their LOC, debt counts, and total `hotspot_score`. This is the "fix here first" table.
4. **Findings table** — every finding: ID, Severity, Category, File:Line, Status, hotspot_score, Title.
5. **Per-finding detail** — heading `### <ID> — <Category> — <Title>`:
   - Severity badge (`panel-info` Info/Low, `panel-warning` Medium, `panel-error` High/Critical).
   - Description + maintainability impact.
   - **Before** code block (or "n/a — not a single-location finding" for duplication groups; for those list every location).
   - **After** code block (or `n/a — needs-refactor`).
   - Status: `✅ auto-fixed` | `🟡 needs-refactor` | `🔴 still-debt` | `⚪ skipped-dirty`.
6. **Cross-link to security** — if any pattern from Phase 1 also smelled like a security issue (e.g. `eval(`, hardcoded URL with auth, deep nesting around auth), list them with the note "Run `/vulnerability-check` to confirm and remediate."
7. **Recommendations** — quarterly re-run; CI integration (`eslint --max-warnings 0`, `coverage --fail-under 70`, `dependabot`); when the team should pick a debt-day; suggested first 3 refactors to schedule.
8. **Run metadata** — project name, scan date, file count, finding count, who ran (`AI tech-debt-check skill`), `git=yes|no`, the wega2/Claude session id if available.

Discover the target Confluence space the same way `sdlc-planning` does — first personal space by default (Shape A: `mcp__Confluence__conf_get` on `/wiki/api/v2/spaces?type=personal`; Shape B: `getConfluenceSpaces`).

Build the URL: `https://<ATLASSIAN_SITE_NAME>.atlassian.net/wiki` + the response's `_links.webui` (Shape A) or `<_links.base><_links.webui>` (Shape B).

## Phase 6 — Final output (mandatory shape)

Print, in this order, exactly:

```
✅ Tech-debt scan complete.

Findings: <N total>  (<c> Critical, <h> High, <m> Medium, <l> Low, <i> Info)
Auto-fixed:     <fixed-count>
Needs refactor: <refactor-count>
Still-debt:     <remaining-count>
Skipped (dirty): <skipped-count>

Hotspot top-5:
  • <file> — <commits_90d> commits/90d, <loc> LOC, score <s>
  • ...

Files modified: <count>
Files modified list:
  • <relative path>
  • ...

✅ Tech-debt report published: <confluence-url>
```

If Confluence publish failed, replace the last line with:
```
🟡 Confluence publish failed (<error>). Report saved locally: <path-to-md-file>
```

If no Atlassian MCP was loaded at all, the local-markdown line is the success line — keep the leading ✅.

## Behavior rules

- **Be autonomous.** No mid-run questions. Only acceptable halts: cwd doesn't exist; zero findings (print a clean "no significant debt found" summary and skip publish); applying a fix would clobber uncommitted edits (skip *that* fix only, log as `skipped-dirty`, continue).
- **Don't conflate with security.** If a finding could be both maintainability AND a vulnerability, prefer **excluding** it here (security has the harder remediation gate) and call it out in the report's cross-link section instead.
- **Don't introduce new dependencies** — neither for analysis (use grep/Read where possible; only invoke `depcheck`/`npm outdated` if they're on PATH) nor for fixes.
- **Don't reformat unrelated regions.** Each `Edit` should be the minimal change for that finding. Leave the file syntactically valid; preserve existing lint/format around the edit.
- **Idempotent.** A re-run on a clean codebase produces a zero-finding report.
- **Never mass-delete code without trace.** Dead-code removal in Phase 3 only touches: unused imports, unreachable lines after `return`/`throw`. Larger dead-code findings (unused exports, large commented blocks) are reported as `needs-refactor` even when they're "obviously" deletable, because a human review on those is cheap and the cost of accidentally removing a still-referenced symbol is high.
