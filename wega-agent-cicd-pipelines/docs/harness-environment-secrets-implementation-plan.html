1. Executive Summary

Key Design Decisions:
  • Environment Variables → YAML file in each service repo (developer-maintained)
  • Secrets → Cloud secret managers (one-time setup, referenced by name)
  • No Code Changes → Applications read via os.getenv() as usual

Why YAML in Each Repo?

  • ✅ Developers own their service's configuration
  • ✅ Changes go through PR review
  • ✅ Version controlled with the code
  • ✅ Easy to understand and maintain
  • ✅ Environment-specific overrides

3.3 Configuration Merge Order

Pipeline merges configs in this order (later overrides earlier):

config/common.yaml → config/{cloud}/common.yaml → config/{cloud}/{env}.yaml + secrets.yaml
Step What Happens
1 Pipeline clones service repo (e.g., wega-brd)
2 Pipeline reads config/common.yaml (global defaults)
3 Merges config/{cloud}/common.yaml (cloud-specific defaults)
4 Merges config/{cloud}/{env}.yaml (environment-specific)
5 Maps secret names from config/secrets.yaml to cloud secret manager
6 Deploys with combined env vars + secrets

4. Secrets Storage (One-Time Setup)

Secrets are stored ONCE in cloud secret managers.
The env-config.yaml only contains the names of secrets, not the values.

4.1 Naming Convention

Cloud Secret Name Format Example
GCP wega-{env}-{secret-name} wega-dev-db-password
Azure Vault: wega-kv-{env}, Secret: {name} wega-kv-dev/db-password
AWS wega/{env}/{secret-name} wega/dev/db-password

4.2 Bulk Secrets Setup (One-Time)

Scripts are provided in scripts/secrets-setup/ to create all secrets at once:

scripts/secrets-setup/ ├── secrets-values.yaml ← Fill in ALL secret values here ├── setup-gcp-secrets.sh ← Run once for GCP ├── setup-azure-secrets.sh ← Run once for Azure ├── setup-aws-secrets.sh ← Run once for AWS └── README.md

Step 1: Fill in secrets-values.yaml

    OPENAI_API_KEY: "sk-actual-key-here"

Step 2: Run Setup Scripts

# GCP - Creates secrets in GCP Secret Manager
./scripts/secrets-setup/setup-gcp-secrets.sh wega-gcp-project

# Azure - Creates Key Vaults and secrets
./scripts/secrets-setup/setup-azure-secrets.sh wega-azure-rg eastus

# AWS - Creates secrets in AWS Secrets Manager
./scripts/secrets-setup/setup-aws-secrets.sh us-east-1

Step 3: Delete secrets-values.yaml

# IMPORTANT: Delete immediately after running scripts!
rm scripts/secrets-setup/secrets-values.yaml

# Or secure delete
shred -u scripts/secrets-setup/secrets-values.yaml
⚠️ Security:
  • secrets-values.yaml is in .gitignore - will not be committed
  • Delete the file immediately after running the scripts
  • Only DevOps/Platform team should have access to actual values
  • Store original values in a password manager (e.g., 1Password, Vault)

5. Pipeline Integration

5.1 Cloud Run Pipeline

# In pipeline-repo-worker-build-deploy-cloudrun.yaml
- step:
    type: Run
    name: Deploy_To_CloudRun
    spec:
      shell: Bash
      command: |
        ENV="<+pipeline.variables.environment>"
        SERVICE_NAME="${ENV}-<+matrix.repo.cloudRunService>"
        REPO_PATH="<+pipeline.variables.repoPath>"
        
        # Merge configs: common → gcp/common → gcp/{env}
        COMMON=$(yq -r '.env // {}' "$REPO_PATH/config/common.yaml")
        CLOUD_COMMON=$(yq -r '.env // {}' "$REPO_PATH/config/gcp/common.yaml")
        ENV_SPECIFIC=$(yq -r '.env // {}' "$REPO_PATH/config/gcp/${ENV}.yaml")
        
        # Combine all env vars
        ALL_VARS=$(echo "$COMMON" "$CLOUD_COMMON" "$ENV_SPECIFIC" | yq -s 'add | to_entries | map("\(.key)=\(.value)") | join(",")' -)
        
        # Parse secrets
        SECRETS=$(yq -r '.secrets[].envVar' "$REPO_PATH/config/secrets.yaml" | while read secret; do
          echo "${secret}=wega-${ENV}-$(echo $secret | tr '[:upper:]' '[:lower:]' | tr '_' '-'):latest"
        done | paste -sd ',' -)
        
        # Deploy
        gcloud run deploy "$SERVICE_NAME" \
          --image="$IMAGE" \
          --region="$REGION" \
          --project="$PROJECT" \
          --platform=managed \
          --set-env-vars="$ALL_VARS" \
          --set-secrets="$SECRETS" \
          --quiet

5.2 ACA Pipeline

# In pipeline-repo-worker-build-deploy-aca.yaml
- step:
    type: Run
    name: Deploy_To_ACA
    spec:
      shell: Bash
      command: |
        ENV="<+pipeline.variables.environment>"
        SERVICE_NAME="${ENV}-<+matrix.repo.acaServiceName>"
        VAULT_NAME="wega-kv-${ENV}"
        REPO_PATH="<+pipeline.variables.repoPath>"
        
        # Merge configs: common → azure/common → azure/{env}
        ALL_VARS=$(yq -s 'add' \
          "$REPO_PATH/config/common.yaml" \
          "$REPO_PATH/config/azure/common.yaml" \
          "$REPO_PATH/config/azure/${ENV}.yaml" \
          | yq '.env | to_entries | map("\(.key)=\(.value)") | .[]')
        
        # Build env vars args
        ENV_ARGS=$(echo "$ALL_VARS" | xargs -I {} echo "--set-env-vars {}")
        
        # Build secrets args (Key Vault references)
        SECRET_ARGS=$(yq -r '.secrets[].envVar' "$REPO_PATH/config/secrets.yaml" | while read secret; do
          SECRET_NAME=$(echo $secret | tr '[:upper:]' '[:lower:]' | tr '_' '-')
          echo "--secrets ${SECRET_NAME}=keyvaultref:https://${VAULT_NAME}.vault.azure.net/secrets/${SECRET_NAME},identityref:system"
        done)
        
        az containerapp update \
          --name "$SERVICE_NAME" \
          --resource-group "$RESOURCE_GROUP" \
          $ENV_ARGS \
          $SECRET_ARGS

5.3 ECS Pipeline

# In pipeline-repo-worker-build-deploy-ecs.yaml
- step:
    type: Run
    name: Generate_Task_Definition
    spec:
      shell: Bash
      command: |
        ENV="<+pipeline.variables.environment>"
        REPO_PATH="<+pipeline.variables.repoPath>"
        
        # Merge configs: common → aws/common → aws/{env}
        ENV_JSON=$(yq -s 'add | .env | to_entries | map({name: .key, value: .value})' \
          "$REPO_PATH/config/common.yaml" \
          "$REPO_PATH/config/aws/common.yaml" \
          "$REPO_PATH/config/aws/${ENV}.yaml" -o json)
        
        # Generate secrets array for task definition
        SECRETS_JSON=$(yq -r ".secrets | map({name: .envVar, valueFrom: \"arn:aws:secretsmanager:us-east-1:${AWS_ACCOUNT}:secret:wega/${ENV}/\" + (.envVar | ascii_downcase | gsub(\"_\"; \"-\"))})" \
          "$REPO_PATH/config/secrets.yaml" -o json)
        
        # Generate task definition
        cat > task-definition.json << EOF
        {
          "family": "${SERVICE_NAME}",
          "containerDefinitions": [{
            "name": "app",
            "image": "$IMAGE",
            "environment": $ENV_JSON,
            "secrets": $SECRETS_JSON
          }]
        }
        EOF
        
        aws ecs register-task-definition --cli-input-json file://task-definition.json

7. Developer Guide

7.1 Adding a New Environment Variable

ScopeFile to Edit
All clouds, all environmentsconfig/common.yaml
One cloud, all environmentsconfig/{cloud}/common.yaml
One cloud, one environmentconfig/{cloud}/{env}.yaml
# config/common.yaml - Add for all clouds/envs
env:
  NEW_FEATURE_FLAG: "true"

# config/gcp/prod.yaml - Override for GCP production only
env:
  NEW_FEATURE_FLAG: "false"

7.2 Adding a New Secret

  1. Request secret creation in cloud secret managers (DevOps team)
  2. Add secret reference to config/secrets.yaml
  3. Pipeline will inject at deploy time
# config/secrets.yaml
secrets:
  - envVar: DB_PASSWORD
  - envVar: NEW_API_SECRET    # ← Add reference here

7.3 Accessing in Code

# Python - No changes needed!
import os

db_password = os.getenv('DB_PASSWORD')
log_level = os.getenv('LOG_LEVEL')
new_secret = os.getenv('NEW_API_SECRET')
Summary:
  • Env Vars → config/ folder in each repo (developer-maintained)
  • Structure → config/{cloud}/{env}.yaml for cloud & environment specific
  • Secrets → Cloud secret managers (one-time setup, DevOps-managed)
  • Pipeline → Merges YAML files, maps secrets, deploys with combined config
  • Code → No changes, continues using os.getenv()
