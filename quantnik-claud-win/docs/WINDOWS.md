# Quantnik — Windows IIS deployment

Deploy Quantnik on a Windows Server (e.g. AWS EC2 Windows) with IIS as the
public-facing edge and Node.js running as a Windows service in the background.

```
Internet → IIS (80/443) → URL Rewrite + ARR → http://127.0.0.1:6060 (Node service)
                       ↘ frontend\dist (static SPA)
```

## Prerequisites on the server

Install these once, as Administrator (PowerShell):

```powershell
# 1. Node.js LTS (20+) — use winget or the official MSI.
winget install OpenJS.NodeJS.LTS

# 2. IIS Web Server role + WebSocket protocol feature.
Install-WindowsFeature -Name Web-Server,Web-WebSockets -IncludeManagementTools

# 3. URL Rewrite module (https://www.iis.net/downloads/microsoft/url-rewrite)
# 4. Application Request Routing (https://www.iis.net/downloads/microsoft/application-request-routing)
#    After install, open IIS Manager → Application Request Routing Cache →
#    Server Proxy Settings → tick "Enable proxy".

# 5. git
winget install Git.Git
```

Reboot if Node or IIS asks for it.

## One-command deploy

```powershell
# Clone the repo to C:\quantnik (or anywhere)
git clone https://github.com/Quantnik-SDLC/quantnik.git C:\quantnik
cd C:\quantnik

# Configure environment (see "Environment variables" below) before running deploy
copy backend\.env.example backend\.env
notepad backend\.env

# Run the deploy script as Administrator
powershell -ExecutionPolicy Bypass -File .\scripts\deploy-windows.ps1
```

`scripts\deploy-windows.ps1`:
1. `npm install` for both backend and frontend.
2. Builds the frontend (`npm run build`) — vite copies `frontend\public\web.config` into `frontend\dist\`.
3. Installs `node-windows` and registers the **Quantnik** service.
4. Prints the remaining manual IIS configuration steps.

## Manual IIS configuration (5 minutes, one-time)

1. **Open IIS Manager.**
2. **Add Website** (or edit Default Web Site):
   - **Site name:** `quantnik`
   - **Physical path:** `C:\quantnik\frontend\dist`
   - **Binding:** http (port 80) — add HTTPS after testing.
3. **Verify URL Rewrite + ARR are installed.**
   IIS Manager → Application Request Routing Cache → Server Proxy Settings →
   tick **Enable proxy**, click Apply.
4. **Verify the WebSocket Protocol feature** is installed
   (`Install-WindowsFeature Web-WebSockets`).
5. **Browse the site** — the Quantnik UI loads at `http://<server-hostname>/`.

## Environment variables

Set these as **System environment variables** (System Properties → Environment
Variables → System variables) so the Windows service inherits them on
restart. After changing them, restart the service:

```powershell
Restart-Service Quantnik
```

| Variable | Required | Notes |
|---|---|---|
| `PORT` | no (default 6060) | Backend port. `web.config` proxies to this. |
| `PROJECTS_ROOT` | yes for persistence | `C:\quantnik-data\projects` recommended. |
| `DB_PATH` | yes for persistence | `C:\quantnik-data\quantnik.db` recommended. |
| `CLAUDE_CODE_OAUTH_TOKEN` | yes (if no API key) | Generated by `claude setup-token` on this server. |
| `ANTHROPIC_API_KEY` | yes (if no OAuth token) | Standard API key from console.anthropic.com. |
| `MCP_*_TOKEN` | optional | Auto-wire MCP servers — see `backend\.env.example`. |

### Generating `CLAUDE_CODE_OAUTH_TOKEN` on the server

```powershell
# Install Claude Code CLI globally
npm install -g @anthropic-ai/claude-code

# Run setup-token (opens a browser — do this in an RDP session, not headless)
claude setup-token

# Copy the printed token and set it as a System env var; then restart the service.
```

## Skills

Skills live in `%USERPROFILE%\.claude\skills\<skill-name>\SKILL.md`. The
repo ships some user-level skills in `.claude-user\skills\`. Copy them
to the runtime user's profile (the account the Quantnik service runs as):

```powershell
$dest = "$env:USERPROFILE\.claude\skills"
New-Item -ItemType Directory -Force $dest | Out-Null
Copy-Item -Recurse .claude-user\skills\* $dest -Force
```

If the Quantnik service runs as `LocalSystem` (the default node-windows
behavior), `%USERPROFILE%` resolves to `C:\Windows\System32\config\systemprofile`.
You can either copy skills there too, or run the service as a named user.
The latter is recommended — IIS → service account → predictable profile path.

## Persistent data directory

Create the folder before first run:

```powershell
New-Item -ItemType Directory -Force C:\quantnik-data\projects | Out-Null
```

Back it up regularly (`robocopy C:\quantnik-data D:\backups\quantnik-data /MIR`
on a scheduled task is fine for a demo).

## Logs

`node-windows` writes service stdout/stderr here (one file per restart):

```
%APPDATA%\node-windows\Quantnik\daemon\Quantnik.out.log
%APPDATA%\node-windows\Quantnik\daemon\Quantnik.err.log
```

When the service runs as `LocalSystem`, that path resolves to
`C:\Windows\System32\config\systemprofile\AppData\Roaming\node-windows\Quantnik\daemon\`.

## Uninstalling

```powershell
node scripts\uninstall-windows-service.cjs
```

## Common problems

| Symptom | Cause / fix |
|---|---|
| `502 Bad Gateway` from IIS | Node service isn't running, or ARR proxy isn't enabled. `Get-Service Quantnik`; check ARR → Server Proxy Settings → Enable proxy. |
| WebSocket chat hangs | WebSocket Protocol feature isn't installed, or `<webSocket enabled="true" />` is missing from `web.config`. |
| `database is locked` | Multiple Node processes hitting the same SQLite. Confirm only one `Quantnik` service exists; kill orphans (`taskkill /F /IM node.exe`) and restart. |
| Skills tab empty | Service runs as `LocalSystem`; skills aren't in its profile dir. Copy them or change the service account. |
| MCP servers empty | Existing project — recreate. Or POST to `/api/mcp/:projectId` to add them manually. |
