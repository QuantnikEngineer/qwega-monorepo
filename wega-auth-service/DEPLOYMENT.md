## Harness Pipeline

Trigger on push. Three steps: build → migrate → deploy. Each service is its own repo and pipeline on Harness — the monorepo is local-only for development convenience.

> **VPC-SC note**: `gcloud builds submit` fails to stream logs due to VPC Service Controls.
> Add `--gcs-log-dir=gs://${PROJECT}_cloudbuild/logs` to redirect logs to a GCS bucket.
>
> **Env var retention**: After the first deploy sets env vars, subsequent image-only redeploys
> (from Harness) preserve them. You only need to set env vars on the first deploy.

```bash
# 1. Build (with VPC-SC log redirect)
gcloud builds submit --tag gcr.io/${PROJECT}/${SERVICE_NAME} \
    --gcs-log-dir=gs://${PROJECT}_cloudbuild/logs

# 2. Migrate (Cloud Run Job — runs before service deploys)
#    Uses the same image. Fails fast (max-retries 0) — if migration fails,
#    deployment aborts and the old service revision stays active.
gcloud run jobs create ${SERVICE_NAME}-migrate \
    --image gcr.io/${PROJECT}/${SERVICE_NAME} \
    --region us-central1 \
    --command python --args=-m,alembic,upgrade,head \
    --task-timeout 300 --max-retries 0 --memory 512Mi \
    --set-env-vars DATABASE_URL=<POSTGRES_CONNECTION_STRING>
gcloud run jobs execute ${SERVICE_NAME}-migrate --region us-central1 --wait

# 3. Deploy (migrations already complete — service skips them)
gcloud run deploy ${SERVICE_NAME} \
    --image gcr.io/${PROJECT}/${SERVICE_NAME} \
    --platform managed --region us-central1 \
    --allow-unauthenticated \
    --memory 1Gi --cpu 1 --concurrency 10 \
    --set-env-vars APP_ENV=production,DEBUG=false,LOG_LEVEL=INFO \
    --set-env-vars COOKIE_SECURE=true,COOKIE_SAMESITE=strict,COOKIE_PATH=/auth \
    --set-env-vars CORS_ORIGINS=<FRONTEND_URL>,FRONTEND_URL=<FRONTEND_URL> \
    --set-env-vars JWT_ISSUER=wega-auth,JWT_AUDIENCE=wega-api \
    --set-env-vars DATABASE_URL=<POSTGRES_CONNECTION_STRING> \
    --set-env-vars INTERNAL_API_KEY=<SHARED_KEY> \
    --set-env-vars WEGA_SECRET_KEY=<FERNET_KEY> \
    --set-env-vars SKIP_MIGRATIONS=true
```

> **Why a separate migration job?** Running migrations inside `entrypoint.sh` on every container
> startup caused: (a) 5-30s startup latency, (b) race conditions when Cloud Run scales multiple
> instances, (c) half-migrated DB if the migration times out. The job pattern ensures migrations
> complete successfully _before_ new code deploys. If the job fails, the old revision stays active.
>
> **Local dev**: The entrypoint still runs migrations automatically when `K_SERVICE` is not set
> (i.e., `docker run` or `docker-compose`). No manual steps needed for local development.

## Deployment Learnings & Gotchas

1. **CRLF → LF**: Shell scripts (`deploy.sh`, `entrypoint.sh`) MUST have Unix LF line endings. Windows CRLF causes `/bin/sh: ... not found` container crashes. Solution: `.gitattributes` with `*.sh text eol=lf`.
2. **VPC-SC Cloud Build**: `gcloud builds submit` fails to stream logs due to VPC Service Controls. Solution: `--gcs-log-dir=gs://${PROJECT}_cloudbuild/logs` redirects logs to a GCS bucket.
3. **Cloud Run retains env vars**: After the first deploy sets env vars, subsequent image-only redeploys (from Harness) preserve them. Only need to set env vars on the first deploy.
4. **Database is always PostgreSQL**: localhost for local dev, AWS RDS for Cloud Run. The `deploy.sh` reads `DATABASE_URL` from `.env` or environment.
5. **Deployment order matters**: `wega-auth-service` → `wega-api-gateway` → `wega-frontend`. Each depends on the previous.
6. **Harness pipelines are separate**: Each service is its own repo and pipeline on Harness. The monorepo is local-only for development convenience.
7. **`init_db()` + Alembic**: `init_db()` creates tables via `create_all`. Alembic migrations seed reference data (org, roles, superadmin).
8. **RDS requires SSL**: The `DATABASE_URL` must include `?ssl=require` — RDS's `pg_hba.conf` only allows `hostssl` connections from Cloud Run IP ranges. Without it, connections fail with `no pg_hba.conf entry... no encryption`.
9. **URL-encode special characters in DATABASE_URL**: Passwords with `$`, `:`, `<`, `>`, `|`, `#` etc. must be URL-encoded (e.g., `$` → `%24`, `:` → `%3A`). SQLAlchemy decodes them automatically. Use Python: `from urllib.parse import quote; quote(password, safe='')`.
10. **Traffic pinning after rollback**: If you pin traffic to a specific revision via `gcloud run services update-traffic --to-revisions=REV=100`, subsequent `services update --update-env-vars` creates a new revision but does NOT move traffic. You must explicitly run `gcloud run services update-traffic --to-latest` to route traffic to the new revision.
11. **Git Bash path mangling**: When running `deploy.sh` on Windows Git Bash, paths like `/auth` in `--set-env-vars COOKIE_PATH=/auth` get expanded to `C:/Users/.../Git/auth`. Use `MSYS_NO_PATHCONV=1` prefix or set env vars via `gcloud run services update` from PowerShell instead.

## Trusted Proxy Mechanism

The auth-service trusts gateway-forwarded requests (X-User-* headers) via **two complementary mechanisms**:

### 1. IP-Based Trust (`TRUSTED_PROXY_IPS`)

- Configure via `TRUSTED_PROXY_IPS=127.0.0.1` (default)
- Works for localhost development and sidecar deployments
- Auth-service checks the request source IP against the trusted list

### 2. Header-Based Trust (`X-Internal-Key`)

- The gateway sends an `X-Internal-Key` header on **every** forwarded request
- Auth-service validates it against its own `INTERNAL_API_KEY` env var
- Works for Cloud Run where the gateway's IP is dynamic and unpredictable

### Cloud Run Configuration

On Cloud Run, `INTERNAL_API_KEY` must be a **48-byte cryptographic token** shared between gateway and auth-service:

```bash
# Set on both services — keys MUST match
gcloud run services update wega-auth-service \
  --update-env-vars INTERNAL_API_KEY=<48-byte-token>

gcloud run services update wega-api-gateway \
  --update-env-vars INTERNAL_API_KEY=<48-byte-token>
```

> **Both mechanisms** are checked: a request is trusted if it comes from a trusted IP **or** carries a valid `X-Internal-Key`. For Cloud Run deployments, header-based trust is the primary mechanism.
